02

Installation par OS

Claude Code et les commandes /plugin sont identiques partout. Seuls changent les outils Ops — et, sur Windows, le shell qui exécute les hooks.

zsh — macOS
# Claude Code $ npm install -g @anthropic-ai/claude-code # ou: brew install claude-code # Outils Ops $ brew install git docker k3d kubectl trivy $ brew install --cask dotnet-sdk # .NET 10 # bash est déjà présent → les hooks .sh tournent nativement ✅
bash — Debian / Ubuntu
# Claude Code $ npm install -g @anthropic-ai/claude-code # Outils Ops $ sudo apt update && sudo apt install -y git bash curl docker.io $ curl -s https://raw.githubusercontent.com/k3d-io/k3d/main/install.sh | bash $ sudo snap install kubectl --classic $ sudo apt install -y trivy dotnet-sdk-10.0 # bash natif → hooks .sh OK ✅
PowerShell — Windows
# Claude Code > npm install -g @anthropic-ai/claude-code # Outils Ops via winget > winget install Git.Git Docker.DockerDesktop Kubernetes.kubectl ` AquaSecurity.Trivy Microsoft.DotNet.SDK.10 > choco install k3d # Chocolatey ou binaire GitHub
⚠️ Hooks sur Windows. Les hooks sont des scripts bash (.sh). Installe Git for Windows (fournit bash) ou active WSL2, et lance Claude Code depuis Git Bash / WSL pour que secret-scan, guard-destructive et session-start s'exécutent. En CMD/PowerShell pur, les .sh ne tournent pas.

Vérifier les prérequis (tous OS) : git --version && docker --version && k3d version && kubectl version --client && trivy --version && dotnet --version

04

Scénario de bout en bout

Bob, nouvel arrivant, doit déployer le service catalog et le voir tourner.

0

Démarrage de session session-start

Le hook SessionStart injecte le rappel : PATH du SDK .NET 10, commande de lancement, skills et garde-fous actifs. Bob n'a rien à mémoriser.

1

État des lieux — /ops-doctor

Inspection en lecture seule de docker, k3d, kubectl, trivy, dotnet, du cluster et des pods → tableau ✅/⚠️/❌. Verdict : outils OK, aucun cluster → action recommandée : /ship.

2

Conteneuriser + déployer — /ship catalog

containerize génère le Dockerfile et scanne l'image (Trivy HIGH/CRITICAL) ; k8s-bootstrap crée le cluster k3d, importe l'image, applique k8s/ ; vérification du rollout + curl /health.

3

Garde-fou destructif guard-destructive · ask

Bob veut repartir de zéro : k3d cluster delete. Le hook intercepte, renvoie une décision ask, Claude demande confirmation avant de détruire. Bob valide en connaissance de cause.

4

Pod en échec → agent ops-engineer + k8s-debug-pod

Le pod ordering est en CrashLoopBackOff. L'agent travaille en lecture seule (describe, logs --previous, events), formule une hypothèse (ex. limits.memory trop bas → OOMKilled) et propose un correctif — appliqué seulement après accord.

5

Commit protégé secret-scan · block

Bob a collé un token par mégarde et commit. Le hook détecte le motif dans l'index, bloque le commit (exit 2) et pointe la ligne fautive. Bob bascule sur une variable d'env, recommit → ça passe.

05

Le point clé : skills vs hooks

Le plugin combine la souplesse des skills et la fiabilité des hooks.

Skills · Commands · Agent

// l'IA mobilise quand c'est pertinent
  • Souples et contextuels
  • Déclenchés selon le jugement du modèle
  • Idéal pour la productivité et l'orchestration
  • Mais : c'est le modèle qui décide

Hooks

// le harnais applique, toujours
  • Déterministes, indépendants du « jugement » du modèle
  • secret-scan bloquera toujours un commit avec secret
  • guard-destructive demandera toujours confirmation
  • C'est ce qui rend un garde-fou fiable
